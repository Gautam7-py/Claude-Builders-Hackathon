# DevOps Skill

## Overview
This skill defines how the DevOps agent takes all generated code and makes
it real — initializing a git repository, committing files, pushing to GitHub,
running database migrations, and writing the README. It is the final step
before QA runs. The output is a committed GitHub repository and a
`devops_done.json` confirmation file.

## When to use
- Use when all code generation agents have confirmed completion
- Use when the GitHub repository needs to be created and populated
- Use when database migrations need to be run via Database MCP
- Use when README.md and .gitignore need to be generated

Do NOT use when:
- Application code is still being generated
- The Architect agent is still designing the system
- `devops` appears in `execution_plan.agents_skipped`

---

## Workflow

### Step 1 — Read all context files
Read before taking any action:
- `.claude/context/manifest.json` — app_name, app_domain, core_features
- `.claude/context/architecture.json` — stack, environment_variables, folder_structure
- `.claude/context/backend_done.json` — files_generated, endpoints_implemented
- `.claude/context/frontend_done.json` — files_generated (or skipped status)
- `.claude/context/test_report.json` — summary for README badges (if exists)
- `.claude/context/execution_plan.json` — agents_to_run, agents_skipped

### Step 2 — Generate .gitignore
Write `.gitignore` to `/output/` before any git commands.
Always include this base set plus stack-specific additions:

```
# Dependencies
node_modules/
__pycache__/
*.pyc
venv/

# Environment
.env
.env.local
.env.*.local

# Build outputs
dist/
build/
.next/
out/

# Logs
*.log
npm-debug.log*

# Editor
.DS_Store
.vscode/
.idea/

# Agent context — never commit
.claude/context/
```

Add stack-specific entries based on `architecture.stack`:
- Prisma: `prisma/*.db`
- SQLite: `*.sqlite`, `*.db`
- Python: `*.egg-info/`, `.pytest_cache/`

### Step 3 — Generate README.md
Write `/output/README.md` with these sections in order:

1. **App name and one-line description** — from manifest app_name and app_domain
2. **Tech stack** — bullet list from architecture.json stack fields
3. **Prerequisites** — Node version, Python version, database engine
4. **Installation**:
   ```bash
   git clone <repo_url>
   cd <app_name>
   cp .env.example .env
   # Fill in your environment variables
   npm install        # in both /frontend and /backend
   ```
5. **Environment variables** — table with Key, Description, Example columns
   derived from `architecture.environment_variables`
6. **Running locally** — separate commands for backend and frontend
7. **API endpoints** — table with Method, Path, Auth Required, Description
   derived from `architecture.api_contracts`
8. **Running tests** — `npm test` commands for backend and frontend
9. **Generated by** — credit line: "Generated by Claude Autonomous App Builder"

### Step 4 — Initialize git and run database migrations in parallel

**Git initialization:**
```bash
cd /output
git init
git add .
git commit -m "feat: initial scaffold of <app_name>

Generated by Claude Autonomous App Builder
Stack: <frontend> + <backend> + <database>
Agents run: <comma-separated agent list>
Complexity: <low|medium|high>"
```

**Database migration** (if `architecture.stack.database` is not `"none"`):
Use Database MCP to run the migration file at the path in
`backend_done.migrations_generated[0]`.

Verify each table in `architecture.data_models` was created successfully.
If migration succeeds and complexity is medium or high, seed one example
record per entity for demonstration purposes.

If Database MCP is unavailable:
- Add manual migration instructions to README.md under a
  `### Database setup` heading
- Log the fallback in `devops_done.json` under `warnings`
- Continue — do not block the pipeline

### Step 5 — Create GitHub repository via GitHub MCP
Repository settings:
- **Name:** `manifest.app_name` in lowercase with hyphens
- **Description:** auto-generated: `"<app_domain> — generated by Claude Autonomous App Builder"`
- **Visibility:** public
- **Default branch:** main

If GitHub MCP is unavailable:
- Log `github_repo_url` as `"local_only"` in `devops_done.json`
- Add manual push instructions to README.md:
  ```
  ### Pushing to GitHub
  git remote add origin https://github.com/<username>/<app_name>.git
  git push -u origin main
  ```
- Continue — do not block the QA agent

### Step 6 — Push all files and write confirmation
```bash
git remote add origin <repo_url>
git push -u origin main
```

Write to `.claude/context/devops_done.json`.

## Validation
Before writing `devops_done.json`, verify:
- [ ] `.gitignore` exists and contains `.env` and `.claude/context/`
- [ ] `README.md` exists and has all required sections
- [ ] `.env.example` was NOT committed (it should be committed; `.env` must NOT be)
- [ ] `git log --oneline` shows at least one commit
- [ ] `devops_done.json` has a non-empty `files_committed` list
- [ ] `github_repo_url` is set (or `"local_only"` if GitHub MCP unavailable)

If any check fails, fix it before writing the confirmation file.

---

## Guidelines
- NEVER commit `.env` files — only `.env.example` is committed
- NEVER commit `.claude/context/` — add it to .gitignore before the first commit
- ALWAYS generate README.md before the first git commit
- ALWAYS generate .gitignore before the first git commit
- ALWAYS write `devops_done.json` before exiting
- ALWAYS include the agent credit line in the commit message
- NEVER modify application code generated by other agents
- CRITICAL: If both GitHub MCP and local git fail, halt and escalate to
  the Error Solver agent — do not silently skip the repository setup
