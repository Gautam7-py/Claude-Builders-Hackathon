# Frontend Skill

## Overview
This skill defines the code standards, file generation order, component
patterns, and API hook conventions the Frontend agent must follow when
producing client-side code. Every file generated by the Frontend agent must
conform to these standards. The output is a complete, syntactically valid
React codebase written to the paths defined in `architecture.folder_structure`.

## When to use
- Use when the Frontend agent generates pages, components, or hooks
- Use when the API client, routing config, or Tailwind setup is being created
- Use when Stitch MCP is active and generating UI components

Do NOT use when:
- `execution_plan.agents_skipped` includes `"frontend"` — skip entirely
- The Architect agent is defining the system blueprint
- The Backend agent is generating server-side code
- The Tester agent is writing test files

## Workflow

### Step 1 — Check if frontend is needed
Read `.claude/context/execution_plan.json`. If `"frontend"` appears in
`agents_skipped`, write to `frontend_done.json` with `status: "skipped"`
and stop. Do not generate any code.

### Step 2 — Read inputs
Read before writing any file:
- `.claude/context/architecture.json` — stack, folder_structure,
  api_contracts, data_models, auth strategy
- `.claude/context/backend_done.json` — `endpoints_implemented` list.
  Only call endpoints confirmed as `"implemented"` — never assume others exist.
- `.claude/context/manifest.json` — app_name, core_features
- `.claude/context/execution_plan.json` — complexity, Stitch MCP status

### Step 3 — Generate files in this exact order
Never change the order — each layer depends on the one before it.

1. **`api/client.js`** — axios instance with base URL and interceptors
2. **`hooks/use<Entity>.js`** — one file per entity from data_models
3. **Components** — base UI via Stitch MCP, then manual fallbacks
4. **Pages** — one file per page, assembled from components and hooks
5. **`App.jsx`** — routing setup, connects all pages last

### Step 4 — Generate the API client
Create a single axios instance. All hooks import from this file only:

```javascript
// src/api/client.js
import axios from 'axios';

const client = axios.create({
  baseURL: import.meta.env.VITE_API_URL,
  headers: { 'Content-Type': 'application/json' },
});

// Attach JWT token to every request if present
client.interceptors.request.use((config) => {
  const token = localStorage.getItem('token');
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

export default client;
```

Never hardcode the API URL. Always use `import.meta.env.VITE_API_URL`.

### Step 5 — Generate one custom hook per entity
One file per entity in `architecture.data_models`. Hooks encapsulate all
API calls — pages never call the API client directly.

```javascript
// src/hooks/useUsers.js
import { useState, useCallback } from 'react';
import client from '../api/client';

/**
 * Hook for managing User data and API operations.
 * @returns {{ data, loading, error, fetchAll, create, update, remove }}
 */
export function useUsers() {
  const [data, setData] = useState([]);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState(null);

  const fetchAll = useCallback(async () => {
    setLoading(true);
    setError(null);
    try {
      const res = await client.get('/users');
      setData(res.data.data);
    } catch (err) {
      setError(err.response?.data?.error || 'Failed to fetch');
    } finally {
      setLoading(false);
    }
  }, []);

  const create = useCallback(async (payload) => {
    const res = await client.post('/users', payload);
    return res.data.data;
  }, []);

  const update = useCallback(async (id, payload) => {
    const res = await client.put(`/users/${id}`, payload);
    return res.data.data;
  }, []);

  const remove = useCallback(async (id) => {
    await client.delete(`/users/${id}`);
  }, []);

  return { data, loading, error, fetchAll, create, update, remove };
}
```

### Step 6 — Use Stitch MCP for UI components
If Stitch MCP is active in execution_plan.json, use it to generate:
- Navbar component
- Form components (one per POST endpoint)
- Card components (list item display)
- Table component (data grid)
- Modal component (confirmation dialogs)
- Base Button, Input, Badge components

For each Stitch call provide:
- Component name (PascalCase)
- Props interface with types
- Color scheme: `indigo` primary + `gray` neutral (default)
- Whether dark mode support is needed

If Stitch MCP is unavailable:
- Generate components manually using Tailwind utility classes
- Log each manual component in `frontend_done.json` warnings with
  `"reason": "stitch_unavailable"`
- Continue without blocking the pipeline

### Step 7 — Determine pages from core_features
Derive pages from `manifest.core_features` using this table:

| Feature keyword | Pages to generate |
|-----------------|-------------------|
| auth, login, register | `LoginPage.jsx`, `RegisterPage.jsx` |
| dashboard, stats, overview | `DashboardPage.jsx` |
| list, table, browse | `ListPage.jsx` |
| detail, view, show | `DetailPage.jsx` |
| form, create, edit | `FormPage.jsx` |
| profile, account | `ProfilePage.jsx` |
| settings | `SettingsPage.jsx` |
| admin | `AdminPage.jsx` |

Always generate regardless of features:
- `HomePage.jsx` — landing page
- `NotFoundPage.jsx` — 404 fallback

### Step 8 — Apply component standards
Every component must:
- Handle three states: `loading`, `error`, and the happy path
- Use Tailwind utility classes only — no inline styles, no CSS files
- Export as a named export (not default export for components)
- Have JSDoc on every exported function

Loading and error pattern for every async component:
```jsx
if (loading) return <LoadingSpinner />;
if (error) return <ErrorMessage message={error} />;
if (!data.length) return <EmptyState message="No items found" />;
```

### Step 9 — Generate App.jsx with routing
Use React Router v6. Every page gets a route:

```jsx
// src/App.jsx
import { BrowserRouter, Routes, Route } from 'react-router-dom';
import Navbar from './components/Navbar';
import HomePage from './pages/HomePage';
import NotFoundPage from './pages/NotFoundPage';

export default function App() {
  return (
    <BrowserRouter>
      <Navbar />
      <Routes>
        <Route path="/" element={<HomePage />} />
        <Route path="*" element={<NotFoundPage />} />
      </Routes>
    </BrowserRouter>
  );
}
```

### Step 10 — Write frontend_done.json
Write to `.claude/context/frontend_done.json` after all files are generated.
Verify each file exists before adding it to the output lists.

## Validation
Before writing `frontend_done.json`, verify:
- [ ] Every page in the pages list is routed in `App.jsx`
- [ ] Every hook calls only endpoints in `backend_done.endpoints_implemented`
- [ ] No hardcoded API URLs — all use `import.meta.env.VITE_API_URL`
- [ ] Every async component has loading, error, and empty states
- [ ] No inline styles exist in any component file
- [ ] `package.json` includes all required dependencies

If any check fails, fix it before writing the confirmation file.

---

## Guidelines
- NEVER call an endpoint not confirmed in `backend_done.endpoints_implemented`
- NEVER write backend code in any frontend file
- NEVER use inline styles — Tailwind utility classes only
- NEVER hardcode API URLs, credentials, or environment-specific values
- ALWAYS add loading, error, and empty states to every async component
- ALWAYS follow the file generation order — api → hooks → components → pages → App
- ALWAYS write `frontend_done.json` before exiting
- CRITICAL: Generated JSX must be syntactically valid — no undefined imports,
  broken component references, or missing closing tags
