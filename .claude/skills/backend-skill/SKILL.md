# Backend Skill

## Overview
This skill defines the code standards, file generation order, response
conventions, and validation patterns the Backend agent must follow when
producing server-side code. Every file generated by the Backend agent
must conform to these standards. The output is a complete, syntactically
valid backend codebase written to the paths defined in architecture.json.

## When to use
- Use when the Backend agent generates API routes, controllers, or models
- Use when middleware, database schema, or `.env.example` is being created
- Use when the server entry point (`index.js` or `server.js`) is being written

Do NOT use when:
- The Architect agent is designing the system blueprint
- The Frontend agent is generating UI components or pages
- The Tester agent is writing test files
- `architecture.stack.backend` is `"none"` — skip entirely

## Workflow

### Step 1 — Check if backend is needed
Read `architecture.stack.backend`. If it is `"none"`, write to
`backend_done.json` with `status: "skipped"` and stop. Do not generate
any code.

### Step 2 — Read inputs
Read before writing any file:
- `.claude/context/architecture.json` — stack, folder_structure,
  api_contracts, data_models, environment_variables, auth strategy
- `.claude/context/execution_plan.json` — complexity level

### Step 3 — Generate files in this exact order
Never change the order — each layer depends on the one before it.

1. **Models** — no dependencies, define data shape first
2. **Controllers** — depend on models, contain business logic
3. **Routes** — depend on controllers, define HTTP surface
4. **Middleware** — error handler, logger, CORS, auth (if needed)
5. **Database schema / migrations** — matches models exactly
6. **Entry point** (`index.js`) — mounts all routes and middleware last

### Step 4 — Apply the response shape standard
Use this shape on EVERY endpoint without exception:

```javascript
// Success response
res.status(200).json({ success: true, data: result });

// Error response
res.status(400).json({ success: false, error: "Descriptive message" });

// Not found response
res.status(404).json({ success: false, error: "Resource not found" });
```

Never use a different shape. The Frontend agent and Tester agent write
assertions against exactly `{ success, data }` and `{ success, error }`.

### Step 5 — Apply input validation on every POST and PUT route
Validate before touching any model or database:

```javascript
// Example: validate required fields
const { title, description } = req.body;
if (!title) {
  return res.status(400).json({ success: false, error: "title is required" });
}
```

Validate every field listed in the `request_body` of the matching
`api_contracts` entry. Never skip validation on any POST or PUT route.

### Step 6 — Apply async error handling on every async function
Wrap every async route handler and controller method:

```javascript
// Route handler pattern
router.post('/resource', async (req, res, next) => {
  try {
    const result = await controller.create(req.body);
    res.status(201).json({ success: true, data: result });
  } catch (err) {
    next(err); // always pass to error handler middleware
  }
});
```

Never leave a catch block empty. Always call `next(err)`.

### Step 7 — Generate middleware files
Always generate these four middleware files:

**errorHandler.js** — catches all errors passed via `next(err)`:
```javascript
export const errorHandler = (err, req, res, next) => {
  console.error(err.stack);
  const status = err.status || 500;
  res.status(status).json({ success: false, error: err.message });
};
```

**logger.js** — logs method, path, and status for every request.

**cors.js** — configures CORS using the `cors` package. Origin from
`process.env.CLIENT_URL` with a fallback to `http://localhost:5173`.

**auth.js** — generated ONLY if `architecture.auth.strategy` is not `"none"`.
Validates JWT from Authorization header. Returns 401 if invalid.

### Step 8 — Generate the database schema
Match every entity in `architecture.data_models` exactly:

**If ORM is Prisma** — generate `prisma/schema.prisma`
**If ORM is Mongoose** — generate one schema file per model in `models/`
**If database is SQLite** — generate raw SQL in `db/migrations/001_init.sql`

Every schema must include: primary key, `created_at`, `updated_at`, all
fields from the data model, and all foreign keys for relationships.

### Step 9 — Generate .env.example
Include every variable from `architecture.environment_variables`. Add a
descriptive comment above each variable. Use placeholder values only —
never real credentials.

```
# Database connection string
DATABASE_URL=postgresql://user:password@localhost:5432/appname

# Server port
PORT=3000

# JWT signing secret (generate a random 64-char string for production)
JWT_SECRET=your_jwt_secret_here
```

### Step 10 — Write backend_done.json
Write to `.claude/context/backend_done.json` after all files are generated.
Verify each file exists before adding it to `files_generated`.

## Validation
Before writing `backend_done.json`, verify:
- [ ] Every endpoint in `architecture.api_contracts` has a matching route file
- [ ] Every entity in `architecture.data_models` has a matching model file
- [ ] `.env.example` exists and contains all variables from `architecture.environment_variables`
- [ ] Entry point file exports `app` for testability
- [ ] No hardcoded credentials, ports, or connection strings exist in any file
- [ ] Error handler middleware is mounted last in the entry point

If a file is missing, generate it before marking the step complete.

## Guidelines
- NEVER implement an endpoint not listed in `architecture.api_contracts`
- NEVER write frontend code in any backend file
- NEVER hardcode secrets, ports, database URLs, or API keys
- NEVER leave a catch block empty — always call `next(err)` or log and rethrow
- ALWAYS validate all POST and PUT request fields before processing
- ALWAYS export `app` from the entry point file for testability
- ALWAYS return 404 for not-found cases — never 500
- ALWAYS generate `.env.example` — never skip it regardless of complexity
- CRITICAL: Every generated file must be syntactically valid — no broken
  imports, undefined references, or incomplete function bodies
